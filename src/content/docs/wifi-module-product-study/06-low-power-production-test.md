## 先给结论

低功耗 Wi-Fi 模组不能只看一个睡眠电流。

真正要评估的是：

```text
睡眠电流
唤醒时间
重新联网时间
上传耗时
平均功耗
业务实时性
断线恢复
量产一致性
```

例如 `Deep Sleep 36uA` 很漂亮，但如果每次唤醒都要重新连接 Wi-Fi 和 MQTT，业务延迟和平均功耗未必一定低。

## 常见功耗状态

可以粗略分成：

```text
Active
Standby
Modem Sleep
Light Sleep
Deep Sleep
```

不同厂商命名不同，但核心差异是：

```text
保留多少状态
能否保持网络连接
唤醒后是否要重新初始化
唤醒耗时多长
功耗降到多少
```

面试时可以说：

> 低功耗不是单纯进入最深 sleep，而是根据业务频率选择模式。如果设备需要实时在线，不能随意 deep sleep；如果只是周期上报，可以唤醒、采集、联网、上传、再睡眠。

## 周期上报设备

适合策略：

```text
Deep Sleep
-> RTC / GPIO 唤醒
-> 采集数据
-> 连接 Wi-Fi
-> 连接 MQTT / HTTP
-> 上传数据
-> 关闭连接
-> 再次睡眠
```

适合产品：

```text
温湿度传感器
电池供电状态采集器
低频工业监测点
```

重点指标：

```text
平均电流
唤醒到上传完成时间
失败重试次数
电池寿命估算
```

## 长连接设备

如果产品需要云端随时下发命令，就不能一直 deep sleep。

适合策略：

```text
保持 Wi-Fi 连接
配置 DTIM / listen interval
MQTT keepalive
必要时进入浅睡眠
```

适合产品：

```text
智能插座
门锁网关
工业控制器
需要远程实时控制的设备
```

取舍：

```text
实时性越强，功耗越高。
睡得越深，唤醒和重连越慢。
```

## Wake 引脚怎么用

模组里的 `Wake` 引脚通常用于：

```text
外部 MCU 唤醒 Wi-Fi 模组
传感器中断唤醒
按键唤醒
定时唤醒
```

典型协作：

```text
主控 MCU 平时低功耗运行
有事件时拉 Wake
Wi-Fi 模组醒来
完成联网和上传
上传完成后通知主控
双方再次进入低功耗
```

## 量产测试要测什么

Wi-Fi 模组量产测试不是只看能不能连上路由器。

建议覆盖：

```text
固件版本
MAC 地址
SN / 设备证书
RF 校准数据
扫描 AP
连接指定 AP
RSSI
TCP / MQTT 测试
UART 通信测试
低功耗电流
唤醒功能
GPIO / ADC / I2C / SPI 基础测试
```

产测工具应该输出：

```text
PASS / FAIL
失败原因
测试耗时
设备 SN
固件版本
RSSI
电流数据
```

## 现场稳定性测试

建议做这些场景：

```text
路由器断电再恢复
服务器断开再恢复
弱信号环境
密码错误
DHCP 失败
DNS 失败
MQTT broker 不可达
TLS 证书错误
长时间运行 24h / 72h
低功耗唤醒 1000 次
高低温或供电波动
```

要记录：

```text
重连次数
最长离线时间
平均联网时间
发送失败次数
接收丢包或解析错误
模组重启次数
主控 ringbuf overflow 次数
功耗曲线
```

## 常见问题定位

Wi-Fi 问题要分层定位：

```text
模组未启动：供电、复位、晶振、固件。
UART 不通：波特率、TX/RX、流控、电平。
扫不到 AP：频段、信号、天线、信道。
连不上 AP：密码、加密方式、RSSI、路由器限制。
无 IP：DHCP、网关、地址池。
连不上服务器：DNS、TCP、TLS、端口、防火墙。
业务失败：MQTT 鉴权、topic、payload、心跳。
```

这个分层回答比“可能网络不好”更专业。

## 面试 30 秒总结

> 低功耗 Wi-Fi 设计要同时看睡眠电流、唤醒时间、重连时间和业务实时性。周期上报设备可以 deep sleep 后唤醒上传；长连接设备要在功耗和在线实时性之间取舍。量产和稳定性测试要覆盖固件版本、MAC、RF、RSSI、联网、MQTT、UART、低功耗电流、唤醒、断网恢复和长时间运行。
